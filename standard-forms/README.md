## Why this directory exists

See [ARCHITECTURE.md â†’ Standard-form comparison](../ARCHITECTURE.md#standard-form-comparison)
and [ARCHITECTURE.md â†’ Section-anchor map](../ARCHITECTURE.md#section-anchor-map-deterministic-detector-scoping).

In brief:

- Reviews diff the uploaded counterparty document against the **canonical standard
  form** for the active playbook version.
- Each section in the standard form gets a stable `section_anchor` (e.g. `sec-8`,
  `sec-10-precedence`).  The diff tagger uses these anchors so hard-rejection rules
  scope to the correct clause.
- The Â§10 Miscellaneous heading contains four distinct sub-clauses in the document.
  These must be split into four anchors (`sec-10-notices`, `sec-10-non-exclusive`,
  `sec-10-merger`, `sec-10-precedence`) so the four Â§10 topics don't collide.  This
  splitting **cannot** fall out of a generic paragraph-level parse; it is
  explicitly configured as data in `playbooks/eiaa-v1.0.0.sections.json`'s
  `sub_clause_splits` block (the single shared document heading each group lives
  under, plus the ordered lettered-paragraph markers that partition its body text)
  and executed by the real-`.docx` loader in `scripts/diff_standard_form.py` and
  the real-`.docx` anchor resolver in `scripts/build_anchor_map.py` (issue #200).

## Anchor map artifact

The anchor map records:
- For each anchor: the **heading text** and its **SHA-256 hash** (`heading_hash`).
- The `standard_form_hash`: SHA-256 of the source `.docx`.
- The `anchor_map_hash`: SHA-256 of the canonical anchor-map JSON.

Both `standard_form_hash` and `anchor_map_hash` are recorded in the **release
bundle** (`playbooks/contract-toaster-v*.json` â†’ `playbook.release.standard_form_hash` /
`playbook.release.anchor_map_hash`) so a review can always be traced back to the
exact standard form and anchor map it used.

## Heading-hash drift gate

CI runs `tests/anchor/test_heading_hash_drift.py` on every change to `standard-forms/`
or `playbooks/`.  It:

1. Loads the active anchor map.
2. Compares each `heading_hash` against the corresponding entry in the playbook's
   `anchor_migrations` array.
3. **Fails** if any anchor's heading text changed but no explicit migration record
   covers that change.

This prevents a standard-form revision from silently repositioning anchors without
the playbook being updated to match.

## Builder fixture tests

CI also runs `tests/anchor/test_anchor_map_builder.py` (issue #75).  It verifies
the builder's own contract, independent of the two governance gates above:
- The synthetic `SECTION_CONFIG` fixture produces the expected anchor set,
  including the Â§10 sub-clause splits (`sub_clause_split: true` and the correct
  `parent_section`).
- `heading_hash` is stable across two builds of the same config (same input â†’
  same hashes).
- Real-`.docx` mode resolves every configured anchor to a heading actually
  present in the document (no silent fallback to the config heading). This
  check is skipped, not failed, when `python-docx` is not installed.
- The builder's written artifact is **byte-identical** across two runs on
  identical input â€” not merely a matching `anchor_map_hash` â€” per the
  no-timestamp-in-persisted-content convention (see `scripts/diff_standard_form.py`).

## Form-coverage gate

CI also runs `tests/anchor/test_form_coverage.py`.  It verifies:
- Every anchor present in the anchor map has **exactly one** corresponding topic in
  `playbooks/contract-toaster-v*.json` (matching `section_anchors`).
- Any anchor with no topic is in this anchor map's `coverage_exempt_anchors` array with
  a reviewed rationale in `coverage_exempt_rationales`.  These two fields are **canonical
  in the anchor map** (siblings of `anchors`), not in the playbook â€” an exemption is a
  reviewed property of the standard-form section it describes.
- No stale exemptions (anchors in `coverage_exempt_anchors` that no longer exist in
  the anchor map).
- Every exempt anchor carries a rationale in `coverage_exempt_rationales`.

## Building / updating the anchor map

Run the builder after any change to the standard-form `.docx`:

```bash
# Synthetic mode (no real .docx required â€” uses the section config in the script):
python3 scripts/build_anchor_map.py

# Real .docx mode (requires python-docx):
pip install python-docx
python3 scripts/build_anchor_map.py --docx standard-forms/eiaa-v1.0.0.docx

# Real-docx mode is exercised offline today against the SYNTHETIC placeholder
# (regenerate it after editing playbooks/eiaa-v1.0.0.sections.json):
python3 scripts/generate_synthetic_standard_form.py
python3 scripts/build_anchor_map.py --docx standard-forms/eiaa-v1.0.0.SYNTHETIC.docx --output /tmp/synthetic.anchor-map.json
python3 scripts/diff_standard_form.py --docx standard-forms/eiaa-v1.0.0.SYNTHETIC.docx
```

Record the printed `anchor_map_hash` in the release bundle before activating.

See also [RUNBOOK.md â†’ Revising the standard form](../RUNBOOK.md#revising-the-standard-form)
for the full procedure including anchor migration records.

## CODEOWNERS

Changes to `standard-forms/` require GC and engineering review (see
[../.github/CODEOWNERS](../.github/CODEOWNERS)).  The standard form and its anchor
map are legal-behavior artifacts: a change to either changes what diff the review
pipeline sees and therefore what the model reasons about.
