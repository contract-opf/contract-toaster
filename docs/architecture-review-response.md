# Architecture-review response

Design lives in [ARCHITECTURE.md](../ARCHITECTURE.md).

This document is the formal response to the earlier principal-level architecture review of `contract-toaster`. That review raised **54 findings** across soundness, security, compliance, operations, audit, supply-chain, architecture, and human-factors lenses. Every finding is tracked here to its resolution and to the exact place in the design docs where the fix now lives.

The intent of this matrix is traceability, not re-argument: each row states the finding in brief, its severity, the one- or two-sentence resolution, and a link to the authoritative section. Where a finding's fix spans several docs, the link points to the section that *owns* the decision; the prose there cross-links the rest. As of June 1, 2026, the **prior 54 findings are closed**, but this file is no longer the current issue register. The June 2026 supplemental review lives in [architecture-issue-spotting-2026-06-01.md](architecture-issue-spotting-2026-06-01.md) and remains the active source for newly identified risks until its findings are filed or resolved.

A note on the two decisions the review asked us to *resolve* rather than *fix*:

- **Finding 20 (SQS vs. Step Functions).** Resolved: the API starts Step Functions **directly**; SQS is removed from the review entry path. This is reflected consistently across every doc (system overview, backend, pipeline stack, data flow, cost shape, runbook prerequisites).
- **Finding 39 (environment strategy).** Resolved: production runs in a **separate AWS account** from dev; v1 still avoids an elaborate multi-account org. Reflected in ARCHITECTURE Environments, design-notes, phase-0 issues, and the runbook conventions.
